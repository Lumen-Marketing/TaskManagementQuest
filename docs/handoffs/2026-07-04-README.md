## Hard rules (all streams)

1. **Stay in your worktree.** Never cd into the main checkout or a sibling
   worktree. Never run `git checkout`/`switch` to another branch.
2. **Verify branch before every commit:** `git branch --show-current` must
   print your stream branch. Concurrent sessions have burned us before.
3. **Never push `main`.** Push your branch (`git push -u origin <branch>`),
   open a PR with `gh pr create`. Vercel auto-deploys `main` to PROD; PRs get
   preview deploys — that preview URL is your manual-verification target.
4. **No destructive git** (`reset --hard`, `clean -f`, force-push): the object
   store is shared with two other live sessions.
5. **File ownership is the contract.** Your brief lists owned files and
   hands-off files. If you must touch a hands-off file, keep it additive and
   tiny, and flag it prominently in your PR description.
6. **`taskmanagement.css` (7.4k lines) is the biggest collision risk.** Only
   edit selectors for your own surface. New rules go in one appended block:
   `/* === stream-<X>: <topic> === */`.
7. **Zero-build vanilla SPA.** No frameworks, no build step, no npm runtime
   deps. Views are hand-rolled classes in `js/views/`.
8. **Mobile first is the standing priority.** Verify every UI change at
   ≤720px viewport before calling it done.
9. **Design language:** warm flat "panze" style — orange `#ED4E0D`, Hanken
   headings, color + contrast instead of borders. NO hairline borders, no
   generic AI-slop patterns.
10. **`env.json` is gitignored and absent locally.** `npm run dev` serves the
    site but Supabase config needs env.json. Do NOT wire PROD credentials into
    local e2e (real company data lives there). Ask the user if you truly need
    local creds; otherwise verify on the PR preview deploy.

## Merge order

**A → B → C.** A is small and surgical; it lands first. B and C: after any
stream ahead of you merges, `git fetch && git rebase origin/main` before
continuing. C carries the largest CSS diff and lands last.
